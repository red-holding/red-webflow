# Анализ PageSpeed (2026-03-15)

Проект: `red-property.com`  
Цель: понять, почему низкий скоринг на мобильных устройствах, и определить конкретные шаги для выхода ПК в зелёную зону (90+).

## Входные данные и методика

- Базовые значения от пользователя: Mobile ~61, Desktop ~89.
- Дополнительная проверка: Lighthouse CLI (`mobile` + `desktop`, категория `performance`).
- Тестовый URL: `https://www.red-property.com/`.

Примечание: абсолютные баллы могут колебаться от запуска к запуску, но узкие места и приоритеты оптимизации остаются стабильными.

## Краткие выводы

- Основные причины просадки mobile: **медленный старт отрисовки**, **высокая нагрузка на main thread**, **большой объём стартового JS и сторонних скриптов**.
- Desktop уже близок к 90+, обычно достаточно устранить **блокирующие стили**, **часть стартовой нагрузки** и **задержку первого экрана из-за прелоадера**.
- Наибольший практический эффект даст уменьшение влияния прелоадера на первую загрузку и отложенная инициализация не критичных ресурсов.

## Первопричины низкого mobile-скоринга

## 1) Прелоадер задерживает реальный первый экран

Текущая логика прелоадера на первой загрузке намеренно удерживает анимацию примерно ~1.5 секунды до полного открытия контента.  
Это напрямую ухудшает FCP/LCP, особенно на мобильных CPU.

## 2) Блокирующий CSS в критическом пути загрузки

На старте блокируют рендер, в частности:

- общий Webflow CSS bundle;
- `preloader/preloader.css`;
- page-specific Webflow CSS;
- `critical.min.css`.

Пока эти стили не загрузятся, браузер откладывает отрисовку.

## 3) Тяжёлые сторонние скрипты на старте

Существенный вклад в неиспользуемый JS и нагрузку main thread дают:

- GTM / GA;
- Yandex Metrika;
- часть служебных библиотек, подключаемых рано.

Это увеличивает parse/eval время до момента интерактивности.

## 4) Тяжёлый сетевой вес первого экрана

Крупные запросы в начале загрузки:

- hero/background медиа (несколько MP4);
- большой variable font файл (~200KB+);
- аналитические скрипты.

Итог: медленнее визуальный старт, особенно в мобильных сетях и на слабых устройствах.

## 5) Крупный DOM и дорогие style/layout операции

Проверка показывает большой объём DOM-узлов, что увеличивает стоимость расчёта стилей, layout и paint.

## Конкретные решения (по приоритету)

## Приоритет A (максимальный ROI, делать в первую очередь)

1. **Снизить влияние прелоадера на первую загрузку**
   - Сохранить прелоадер для переходов между страницами.
   - На первой загрузке сильно сократить тайминги или отключить на mobile.
   - Дополнительно: отключать анимацию для `prefers-reduced-motion`.

2. **Уменьшить блокирующий CSS**
   - В `head` оставить только реально критичный above-the-fold CSS.
   - Остальные стили перевести в отложенную стратегию загрузки.
   - Проверить, что стили прелоадера минимальны и не тянут лишнее.

3. **Отложить запуск аналитики**
   - Инициализировать GTM/GA/Yandex после `load` или первого взаимодействия.
   - Не запускать весь аналитический стек до первой осмысленной отрисовки.

## Приоритет B (сильный эффект)

4. **Оптимизировать загрузку hero-медиа**
   - Сначала показывать лёгкий poster/изображение.
   - Видео запускать после первого paint (или при входе секции во viewport).
   - Для mobile использовать отдельные низкобитрейтные источники.

5. **Условная загрузка скриптов по страницам/секциям**
   - Подключать `swiper`, phone input utilities и другие библиотеки только там, где они реально используются.
   - Убрать глобальную стартовую загрузку feature-скриптов.

6. **Оптимизировать доставку шрифтов**
   - Делать subset по реально используемым glyph/weight above the fold.
   - Использовать `font-display: swap`.
   - Preload только нужных для первого экрана файлов.

## Приоритет C (очистка и стабилизация)

7. **Упростить DOM главной страницы**
   - Убрать лишние обёртки.
   - Отложить рендер не критичных блоков ниже первого экрана.

8. **Снизить конкуренцию скриптов на старте**
   - Перенести не критичные скрипты после основного render path.
   - Удалить дубли и устаревшие утилиты.

## Стратегия выхода ПК в 90+

При текущем уровне desktop (~89) чаще всего хватает трёх действий:

1) уменьшить задержку первого экрана от прелоадера;  
2) сократить блокирующий CSS;  
3) отложить инициализацию аналитики.

Это обычно поднимает desktop в зелёную зону без редизайна.

## Чек-лист ручного внедрения

1. Прелоадер:
   - сократить тайминги первой загрузки;
   - сохранить логику для page-to-page переходов.
2. Head/CSS:
   - проверить все `<link rel="stylesheet">` в `head`;
   - отложить не критичные стили.
3. Аналитика:
   - перенести старт GTM/GA/Yandex на более поздний триггер.
4. Медиа:
   - poster first, отложенный старт видео;
   - проверить размеры/битрейт мобильных источников.
5. Скрипты:
   - подключать feature-библиотеки только на нужных страницах.
6. Шрифты:
   - subset + `swap` + избирательный preload.

## Протокол повторных проверок

- Запускать Lighthouse Mobile и Desktop после каждого приоритетного блока.
- Отслеживать: FCP, LCP, TBT, INP proxy, общий объём JS, время блокирующего CSS.
- Продолжать оптимизацию, пока desktop не стабилизируется на 90+ и mobile не покажет устойчивый рост.
